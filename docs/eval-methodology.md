# 检索质量评测体系:技术与指标总结

> 对象:本项目(Agent 搜索引擎)的 IR 评测体系。
> 代码:[eval/](../eval/) ｜ 数据集:[eval/dataset.jsonl](../eval/dataset.jsonl) ｜ 最新报告:[eval/report.md](../eval/report.md)
> 一句话:**用 LLM 当裁判给「查询-文档」打分级相关性,再用标准 IR 指标量化不同检索配置的排序质量**,让每项改动「在评测集上量化、不凭感觉」。

---

## 1. 为什么要这套评测

搜索/RAG 的改动(换源、加重排、调融合权重)好不好,**人工抽检会骗人**——少量样本上「看着合理」的功能,全量 A/B 可能整体有害。本项目就踩过一次:信号融合在 12 条抽检时显得不错,扩到 30 条全量评测才暴露它拉低 NDCG(详见第 7 节)。

因此需要一套**可复现、可量化、低成本**的离线评测,核心三件事:

1. **评测集**:覆盖多种查询类型的真实问题。
2. **相关性判分**:每个「查询-文档」对该文档有多相关 → 分级标签。
3. **IR 指标**:把「一批排好序的结果 + 相关性标签」压成几个可比的数。

---

## 2. 评测数据流

```
dataset.jsonl (30 条查询)
      │
      ▼
多源检索(腾讯/百度/SerpAPI),结果落盘缓存 ── eval/cache/search_<provider>.json
      │
      ▼
按多种「配置」组装 Top-K              ── 单源 / RRF融合 / SF重排 / SF重排+信号融合
      │
      ├──► 收集判分池 pool:每 query 下「所有配置 Top-K 的并集」
      │         │
      │         ▼
      │    LLM-as-judge(Claude)对 pool 里每个 (query,doc) 打 0–3 分 ── 落盘 judgments.json
      │
      ▼
对每个配置算 NDCG@k / Recall@k / P@k / MRR,跨 query 宏平均
      │
      ▼
配置对照表 → eval/report.md
```

关键设计:**检索结果与判分都落盘缓存**,所以重复跑评测时只对「新查询/新文档」付费,结论可复现。

---

## 3. 评测集(dataset.jsonl)

30 条真实中/英文查询,每条带 `type` 标签,覆盖 **8 类**检索难点:

| 类型 | 条数 | 考察点 | 示例 |
|------|------|--------|------|
| `factual` | 4 | 事实召回、权威页面 | 珠穆朗玛峰的海拔高度是多少 |
| `timely` | 5 | 时效性、新鲜度 | 今天A股大盘行情怎么样 |
| `multihop` | 3 | 多跳/对比推理 | RAG 和模型微调各自的优缺点 |
| `longtail` | 3 | 长尾、专有名词 | BGE-Reranker-v2-m3 模型的特点 |
| `mixed` | 2 | 中英混合术语 | 什么是 Mixture of Experts 架构 |
| `howto` | 3 | 操作指南 | 怎么用 Python 读取一个 JSON 文件 |
| `colloquial` | 4 | 口语化、噪声表达 | 帮我查一下北京有什么好吃的地方 |
| `english` | 3 | 英文/全球覆盖 | best practices for retrieval augmented generation |
| `domain` | 3 | 垂直领域 | 中医针灸治疗失眠的原理和效果 |

> 类型标签让我们能**分桶看效果**——比如发现「新鲜度信号帮了 timely、害了 factual」,而不是只看一个总分。

## 4. 相关性判分:LLM-as-judge([eval/judge.py](../eval/judge.py))

人工标注贵且慢,本项目用 **Claude 当裁判**自动打分级相关性。

### 4.1 分级标准(graded relevance, 0–3)

评分细则写进 system prompt,要求模型**只输出一个数字**:

| 分 | 含义 |
|----|------|
| **3** | 高度相关:直接、充分回答查询核心意图 |
| **2** | 相关:有直接相关的有用信息,但不完整 |
| **1** | 略相关:沾边/提到话题,基本无法用于回答 |
| **0** | 不相关:无关,或广告/导航/无效内容 |

> 分级(而非二值)相关性是 NDCG 能体现「排序精细质量」的前提——好结果排前面 vs 中等结果排前面,分数不同。

### 4.2 工程要点

- **模型**:`claude-haiku-4-5`(便宜快,判分任务足够),`max_tokens=8`,正则抽取 `[0-3]`。
- **Prompt caching**:评分细则放 system 段并开 `cache_control: ephemeral`,大批量判分时复用、压成本。
- **判分缓存**:结果按 `sha1(query + url)` 落盘 `eval/cache/judgments.json`,**保证可复现 + 省调用**(命中缓存不重判)。
- **并发**:`ThreadPoolExecutor(workers=6)` 并行判分。
- 文档文本截断到 2500 字符喂给裁判(控 token)。

### 4.3 Pooling(判分池)

不可能给「全网文档」判分,业界标准做法是 **pooling**:只判**各配置 Top-K 结果的并集**。

```
pool[query] = ⋃  TopK(config, query)      对所有参与对照的配置取并集
```

- IDCG、Recall 的分母都基于这个 pool(近似「该 query 的全部已知相关文档」)。
- 缺点是绝对召回偏乐观(池外的相关文档没被发现),但**配置间横向对比是公平的**——这正是评测要回答的问题。

## 5. 评测指标([eval/metrics.py](../eval/metrics.py))

记 `ranked_rels` = 某配置 Top-K 按排名的相关性分级列表;`pool_rels` = 该 query 判分池里全部文档的分级。二值指标用阈值 **rel ≥ 2** 视作「相关」。

### 5.1 NDCG@k — 排序质量(主指标)

归一化折损累计增益,用**分级相关性 + 指数增益**,对「相关结果排得越靠前」给越高分:

```
DCG@k  = Σ_{i=1..k}  (2^rel_i − 1) / log₂(i + 1)
IDCG@k = pool 内按相关性理想排序后的 DCG@k
NDCG@k = DCG@k / IDCG@k        ∈ [0, 1]
```

- `2^rel − 1` 增益:3 分文档(增益 7)远重于 1 分(增益 1),强调高相关。
- `log₂(i+1)` 折损:越靠后的位置贡献越小,体现「排序」价值。
- 归一化到 0–1,可跨 query 平均。**这是衡量重排好坏最敏感的指标**。

### 5.2 Recall@k — 召回完整性

Top-K 里相关文档数 / 判分池里相关文档总数:

```
Recall@k = |{相关 ∩ TopK}| / |{pool 内全部相关文档}|
```

衡量「该捞的有没有捞回来」。多源聚合的价值主要体现在这——单源 Recall 低,聚合后升高。

### 5.3 Precision@k — 结果纯净度

```
P@k = |{相关 ∩ TopK}| / k
```

Top-K 里有多少是真相关。直接关系喂给 LLM 的上下文有多干净(阈值过滤、去噪影响这一项)。

### 5.4 MRR — 首个相关结果的位置

```
MRR = 1 / (首个相关文档的排名)
```

第一个相关结果排第几。对「秒答场景」重要——用户/agent 往往只看最靠前的。

### 5.5 聚合

每个指标先**逐 query 算**,再对 30 条查询取**宏平均**(macro-average,每条查询等权),得到配置级单值。

## 6. 对照配置与复现([eval/run_eval.py](../eval/run_eval.py))

通过对比一组「配置」回答关键选型问题:**哪个源强?多源值不值?重排值不值?信号融合有没有用?**

| 配置 | 含义 | 回答的问题 |
|------|------|-----------|
| `tencent` / `baidu` / `serpapi` | 单源原始去重 | 各源单独多强 |
| `triple+rrf` | 三源 RRF 融合 | 纯位次融合的多源效果 |
| `triple+sf` | 三源 + SiliconFlow 重排 | 重排的增益(对比 rrf) |
| `triple+sf+fusion` | 三源 + 重排 + 信号融合 | 新鲜度/权威度信号有没有用 |

**复现命令**(项目根目录):

```bash
.venv/bin/python -m eval.run_eval                 # 全量 30 条
.venv/bin/python -m eval.run_eval --max-queries 4 # 快速冒烟
.venv/bin/python -m eval.run_eval --no-judge      # 只看检索延迟,不调裁判
```

输出配置对照表到终端 + [eval/report.md](../eval/report.md)。

## 7. 评测体系带来的关键结论

最新一轮(30 条查询,k=10)的对照:

| 配置 | NDCG@10 | Recall@10 | P@10 | MRR |
|------|---------|-----------|------|-----|
| 腾讯单源 | 0.844 | 0.381 | 0.880 | 0.983 |
| 百度单源 | 0.753 | 0.399 | 0.900 | 0.936 |
| SerpAPI 单源 | 0.482 | 0.241 | 0.633 | 0.704 |
| 三源 + RRF | 0.774 | 0.394 | 0.880 | 0.983 |
| **三源 + SF 重排** | **0.906** | **0.442** | **0.963** | **0.983** |
| 三源 + SF + 信号融合 | 0.896 | 0.442 | 0.963 | 0.983 |

评测体系直接支撑了三个决策:

1. **重排是核心质量杠杆**:`+0.13 NDCG`(0.906 vs RRF 0.774),确立为生产默认。
2. **信号融合是负优化,默认关闭**:0.896 < 0.906。分桶分析显示新鲜度信号误伤了 factual 类的权威老页面(百科等)。**这一结论 12 条抽检时看不出来,扩到 30 条全量 A/B 才暴露。**
3. **SerpAPI 单源弱但聚合有益**:单源仅 0.482,但并入后整体 Recall 仍升,补了英文/全球覆盖。

> **核心方法论**:每项改动都回到这张表上量化,而不是凭人工抽检的「感觉」。这套评测的最大价值,就是把「信号融合」这种看着合理实则有害的功能挡了下来。

---

## 8. 已知局限与后续

- **判分池偏乐观**:Recall 分母基于 pooling,池外相关文档未知 → 绝对值偏高,但横向对比有效。
- **单裁判**:仅 Claude 一个 judge,未做多裁判一致性(如 Cohen's κ)交叉验证 → 后续可加。
- **只评检索段**:当前是检索质量(retrieval IR),未做**端到端 RAG 评测**(faithfulness / context precision)→ 列为后续。
- **规模**:30 条仍偏小,扩到 50–100 条并做人工抽检校准 judge 一致性,可让结论更稳。
